# 个人健康看板 (AI Health Dashboard) 设计文档

## 1. 项目愿景

本项目是一个以 **AI 视觉食物识别** 为核心切入点的自动化个人健康、饮食记账系统。

* **核心痛点**：解决传统方案手动录入繁琐、配料表需自己进行 100g 换算的计算折磨。
* **核心逻辑**：通过 AI 与模板机制自动结算每餐的 macros （蛋白质/脂肪/碳水），结合基础代谢（BMR）计算“实时热量缺口”。
* **扩展性保障**：采用 FastAPI + Vue3 前后端分离，具备完整的 JWT 用户认证链路，支持向多人 SaaS 或移动端套壳平滑升级。

## 2. 核心架构与技术选型

| 模块 | 推荐技术 | 选型动机 |
| --- | --- | --- |
| **后端 API** | **FastAPI (Python 3)** | 异步高并发，生态契合各类 AI SDK，自生成 Swagger 文档。 |
| **持久化层** | **SQLite + SQLAlchemy ORM** | Schema 灵活改动。通过 `user_id` 列实现多用户的逻辑隔离。 |
| **前端展现** | **Vue 3 + Tailwind CSS + Vant 4** | 偏向 Mobile-First 的 UI，提供接近 Native App （例如相机唤醒、时间滚轮）的交互。 |
| **AI 引擎** | **ZhipuAI (智谱清言)** | 本地化快速接入，`glm-4v-flash` 执行摄入物视觉读取，`glm-4-flash` 处理纯文本食材描述提取。 |
| **鉴权认证** | **JWT (JSON Web Tokens)** | 无状态 Token，轻松实现“认牌不认人”的并发接口安防隔离。 |

## 3. 核心功能与心智模型

### 3.1 🧠 智慧摄入系统 (Smart Intake)
饮食记录被设计为三大高效入口：
1. **AI 视觉与语意识别**：用户直接丢入照片，或口语化输入（如“半碗米饭加一盘牛肉”），系统结构化提取对应热量和营养素。
2. **我的常用 (Quick Add)**：用户点击一键记录整份常吃餐品。
3. **配料表模板 (Ingredient Templates)**：
    * **逻辑痛点**：食品包装背后总是标示 `xx大卡/100g` 或 `xx千焦/100g`。
    * **系统解法**：系统底层具备 kJ 自动换算 kcal 的能力。保存配料表数据时，打上 `is_per_100g = True` 标签。用户下次调用该模板时，系统强制唤醒重量滑轮，用户输入如 `25g`，系统将在底层按比例自动分割所有三大营养素和热量存入账单。支持时间戳倒退补录。

### 3.2 📊 动态热量预算与实时缺口 (Dynamic Budgeting)
* **动态 TDEE 结算**：摒弃每天固定死数字的模式。系统内置“练腿 / 胸背 / 有氧 / 休息 / 放纵”等 5 种场景参数因子。
* **预算方程式**：
    * `目标热量 = BMR × 场景系数`
    * `碳水 / 蛋白 / 脂肪`：根据当天训练目标结合用户体重（例如：练腿不设缺口，休息日采取低碳侧重蛋白质修复）。
* **UI 展现**：页面提供全局的水位进度条展示。实时展示 `TDEE - 今日已吃`，低于零时触发红色警报。

## 4. 数据库设计基准 (Entity-Relationship)

**⚠️ 贯穿准则：所有含有隐私的记录表，必定包含 `user_id` 作为隔离防浪堤。**

### 4.1 表: `users` (租户基础)
* `id` (PK)
* `username`, `hashed_password` (存储脱敏秘钥)

### 4.2 表: `user_config` (生理特征)
* `id` (PK), `user_id` (FK) -> 强关联 user，一人一行。
* `bmr_kcal` (基础代谢), `height`, `weight`, `gender`, `activity_multiplier`。

### 4.3 表: `saved_food` (常备粮储)
* `id` (PK), `user_id` (FK), `name`
* `kcal`, `protein` (Float), `fat` (Float), `carb` (Float)
* `is_per_100g`: BOOLEAN （用于标识该记录是否属于“配准表100g切分体系”）

### 4.4 表: `intake_log` (摄入流水)
* `id` (PK), `user_id` (FK)
* `date` (归档日), `time` (具体进食时间)
* `food_name`, `kcal`, `macros` (JSON 格式化封装三大宏量), `img_path` (留存追溯)。

## 5. 云端部署与架构边界

系统目前采用单机内聚架构，配合 Alibaba Cloud (Ubuntu) 实现云端化：
* **边界反向代理**：Nginx 处理 80 端口进入的所有流量。静态 `/` 路由直接伺服 Vue.js 编译生成的 `dist`，动态 `/api/` 反代给后端池。
* **WSGI 守护兵**：Gunicorn + Uvicorn Worker 在后台通过 Systemd 不死进程维持 Python FastAPI 的存活。
* 当前架构采用单 SQLite 支撑，对于开发与小规模亲友内测已具备超额性能。未来若达到瓶颈可只需更换 SQLAlchemy URI 以平滑无损迁移至 PostgreSQL。
